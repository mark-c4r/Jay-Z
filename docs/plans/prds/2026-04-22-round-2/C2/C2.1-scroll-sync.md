## Problem

The page already has a stateful topbar path: `state.currentStop` + `state.currentBlock` feed `updateTopbar()`, and click-driven navigation updates that state before calling `updateTopbar()`. The gap is that scroll position never updates `state.currentStop`, so a reader who lands mid-page or scrolls down sees stale topbar context from the last click/load state.

This PRD adds scroll-sync without inventing a second topbar writer. An `initScrollSync()` block watches the 6 `.stop` sections, updates `state.currentStop`, resets `state.currentBlock` to `1`, then hands all UI writes back to the existing `renderRail()` + `updateTopbar()` flow. A pre-IO fallback (for browsers where `window.IntersectionObserver` is undefined) uses a throttled `scroll` listener + `getBoundingClientRect()`.

**Explicit scope:** no `data-stop-title` markup, no direct writes to `#tb-stop`, `#tb-title`, `#tb-fill`, `#tb-block`, or `#tb-blocks-total` from the new block. The existing `STOPS` array remains the title source of truth.

**Add:**

```javascript
      function initScrollSync() {
        const stops = Array.from(document.querySelectorAll('section.stop[id^="stop-"]'));
        if (stops.length === 0) return;

        const applyStop = (section) => {
          const match = section.id.match(/^stop-(\d+)$/);
          if (!match) return;
          const nextStop = parseInt(match[1], 10);
          if (!Number.isFinite(nextStop) || nextStop === state.currentStop) return;
          state.currentStop = nextStop;
          state.currentBlock = 1;
          saveState(state);
          renderRail();
          updateTopbar();
        };

        const pickMostVisible = () => {
          const vh = window.innerHeight || document.documentElement.clientHeight;
          let best = null;
          let bestRatio = 0;
          for (const section of stops) {
            const r = section.getBoundingClientRect();
            const visible = Math.max(0, Math.min(vh, r.bottom) - Math.max(0, r.top));
            const ratio = visible / Math.max(1, Math.min(vh, r.height));
            if (ratio >= 0.4 && ratio > bestRatio) {
              best = section;
              bestRatio = ratio;
            }
          }
          return best;
        };

        if (typeof window.IntersectionObserver !== 'undefined') {
          const io = new IntersectionObserver((entries) => {
            const active = entries
              .filter(entry => entry.isIntersecting && entry.intersectionRatio >= 0.4)
              .sort((a, b) => b.intersectionRatio - a.intersectionRatio)[0];
            if (active) applyStop(active.target);
          }, { threshold: [0, 0.4, 0.5, 0.75, 1] });
          stops.forEach(section => io.observe(section));
          const initial = pickMostVisible();
          if (initial) applyStop(initial);
        } else {
          let rafId = 0;
          const onScroll = () => {
            if (rafId) return;
            rafId = requestAnimationFrame(() => {
              rafId = 0;
              const active = pickMostVisible();
              if (active) applyStop(active);
            });
          };
          window.addEventListener('scroll', onScroll, { passive: true });
          onScroll();
        }
      }
```

**Diff intent:**
- Use existing `state` + `renderRail()` + `updateTopbar()` instead of writing spans directly.
- Reset `state.currentBlock = 1` on scroll-driven stop changes so the block counter stays sane for the newly active stop.
- Use existing `STOPS` array for short titles; no new markup or title-mirroring attributes.

## Known risks + mitigations

1. **`updateTopbar()` still updates block counters/fill** — this PRD does not write those surfaces directly, but calling the existing writer will refresh them. Acceptable; it preserves a single topbar source of truth.
2. **Multiple stops above the 40% threshold** — pick the most-visible section by intersection ratio. The initial `pickMostVisible()` pass also covers deep-linked/page-restored loads.
3. **Fast scrolling on old browsers** — the fallback throttles with `requestAnimationFrame`. Good enough for 6 sections.
4. **Short topbar titles vs longer H2s** — deliberate. The `STOPS` array remains the topbar label source, which matches the existing rail/topbar copy and avoids the H2-vs-short-label contradiction Codex flagged.
