## Problem

Mobile Safari and Android Chrome sometimes drop the `click` event on YouTube facades on touch, or fire it after a 300ms delay during which the reader taps again. The facade click handler (`index.html:2080`) binds `click` only. Result: first-tap-no-response on mobile, sometimes double-activation if the user retaps.

C5.2 already shipped the structural fix — `!facade.isConnected` early-return in `activateFacade`, preventing duplicate iframes from double-activation. This PRD adds the complementary **input-side fix**: a `touchend` handler that calls `activateFacade(facade)` too, and `preventDefault()` on the touchend to suppress the synthesized click that follows. Both paths now funnel through `activateFacade`, whose guard prevents duplicate iframes.

## Work — exact edits

### Edit 1: Add `touchend` alongside existing `click` handler on facades

**File:** `index.html`
**Anchor:** the `.yt-facade` click-binding loop at around line 2079-2081
**Action:** augment the loop to also bind `touchend`

**Before** (state after C5.2):

```javascript
      // --- YouTube facade: click to swap facade → live iframe with autoplay ---
      document.querySelectorAll('.yt-facade').forEach(facade => {
        facade.addEventListener('click', () => activateFacade(facade));
      });
```

**After**:

```javascript
      // --- YouTube facade: click + touchend to swap facade → live iframe with autoplay (C2.2).
      //     activateFacade is idempotent (!facade.isConnected early-return, from C5.2),
      //     so a touchend followed by a synthesized click is safe.
      document.querySelectorAll('.yt-facade').forEach(facade => {
        facade.addEventListener('click', () => activateFacade(facade));
        facade.addEventListener('touchend', (e) => {
          // preventDefault suppresses the 300ms-delayed synthetic click after touchend.
          e.preventDefault();
          activateFacade(facade);
        }, { passive: false });
      });
```

**Note on `{ passive: false }`:** required because we call `preventDefault()` inside the handler. Without this, modern browsers may warn or ignore the `preventDefault`.

## Known risks + mitigations

1. **Desktop browsers with touchscreen** — Chrome on Windows touchscreens fires both touchend and click. The idempotent guard makes this safe (second call to activateFacade is a no-op).
2. **Accessibility / screen reader** — screen readers simulate click, not touchend. The existing click binding still fires, so SR-activation continues to work.
3. **preventDefault on touchend may block scroll** — only fires on the facade element, which is a `<button>` with no scroll behavior. Safe.
4. **`{ passive: false }` performance cost** — minor; only 7 facades. Not a meaningful hot path.
