## Problem

C5.3 rendered `<button class="video-chapter-btn" data-seek="<int>">` under each facade. Clicking one currently does nothing. This PRD wires the click: if the facade is still static (iframe not bootstrapped), activate it with `startOverride` = `data-seek` via C5.2's extended signature; if the iframe is already active, post a YouTube `seekTo` via `postMessage` (modeled on the existing `.highlights-list a[data-seek]` handler).

**Design — two-branch click handler:**

```
click on .video-chapter-btn
  ├── find closest parent with .block-grid or similar video container
  ├── is there a .yt-facade in that container? → branch A: activateFacade(facade, seek)
  └── else is there an iframe? → branch B: postMessage seekTo (+scrollIntoView)
```

Branch A uses C5.2's signature. Branch B uses the same postMessage pattern the existing `.highlights-list` handler uses. A third fallback (both missing — unlikely) silently no-ops.

```javascript
      renderVideoChapters();

      // --- Chapter button click handler (C5.4): branch on facade-vs-iframe.
      document.querySelectorAll('.video-chapter-btn').forEach(btn => {
        btn.addEventListener('click', (e) => {
          e.preventDefault();
          const seek = parseInt(btn.dataset.seek, 10);
          if (!Number.isFinite(seek)) return;
          const grid = btn.closest('.block-grid');
          if (!grid) return;
          const facade = grid.querySelector('.yt-facade');
          if (facade) {
            // Branch A: facade still static — activate with chapter start as override.
            activateFacade(facade, seek);
            return;
          }
          const iframe = grid.querySelector('iframe');
          if (!iframe || !iframe.contentWindow) return;  // silent no-op fallback
          // Branch B: iframe live — seek via YouTube IFrame API postMessage.
          try {
            iframe.contentWindow.postMessage(JSON.stringify({
              event: 'command',
              func: 'seekTo',
              args: [seek, true]
            }), '*');
            iframe.scrollIntoView({ behavior: 'smooth', block: 'center' });
          } catch (err) {
            /* defensive: if postMessage fails, do nothing; fallback is the native anchor, but we preventDefault'd */
          }
        });
      });
```

## Known risks + mitigations

1. **Dynamically-injected buttons (C5.3's `renderVideoChapters`) — timing race** — `renderVideoChapters()` runs synchronously in the main IIFE, and this handler's `document.querySelectorAll('.video-chapter-btn')` runs immediately after. The buttons exist by the time `forEach` runs. Safe.
2. **Re-renders** — if any future code calls `renderVideoChapters()` a second time, newly-created buttons won't have handlers attached. Mitigation: the current code path calls `renderVideoChapters()` exactly once. If C5.5 or later needs re-render, that PRD adds event delegation or a re-bind helper.
3. **postMessage origin** — we pass `'*'` as target origin. Matches existing `.highlights-list` pattern. Safer would be `'https://www.youtube.com'`, but changing one without the other creates inconsistency; out of scope for this PRD. Optional Codex note.
4. **Chapter click while iframe is loading** — if the iframe exists but `contentWindow` isn't yet ready (bootstrap delay), the `!iframe.contentWindow` guard returns. Reader has to click again. Small UX annoyance; safer than crashing. If Codex flags, add a retry with `setTimeout`; track as risk.
