## Problem

With `videoChapters` data embedded (C5.1) and the extended `activateFacade` signature (C5.2), this PRD adds a visible **chapter list** beneath each of the 7 YouTube facades. Readers see interval-overlap-filtered chapters and can click to seek. Interaction wiring is C5.4; THIS PRD is render + CSS only.

Rendering happens at page-load time via a small JavaScript function that iterates `.yt-facade` elements, applies the interval-overlap filter against `videoChapters`, and injects a `<ul class="video-chapters">` immediately after each facade. Chapter titles are rendered inside `<button type="button" data-seek="<clamped-start>">` elements with "MM:SS — Title" format.

**Clamping rule:** if a chapter's `start` is before the facade's window start, the button's `data-seek` equals `facadeStart` (not the chapter's true start). This prevents seeking into pre-facade content when the reader clicks an early chapter. Title display preserves the true chapter title.

Displayed `MM:SS` text uses that same clamped seek value. This is deliberate: the button label mirrors the actual seek target, not the source chapter's pre-clamp start.

**CLS reservation:** the `<ul class="video-chapters">` has `min-height` sized for the maximum expected chapter count (3, per facade-interval-overlap math) to prevent layout shift when iframes bootstrap below.

      document.querySelectorAll('.yt-facade').forEach(facade => {

      });

      // --- Chapter lists (C5.3): render <ul class="video-chapters"> below each
      //     facade, filtered by interval-overlap against the facade window.
      //     Click wiring lives in C5.4.
      renderVideoChapters();

      function renderVideoChapters() {
        document.querySelectorAll('.yt-facade').forEach(facade => {
          const videoId = facade.dataset.videoId;
          const facadeStart = parseInt(facade.dataset.start, 10);
          const facadeEnd = parseInt(facade.dataset.end, 10);
          const allChapters = (videoChapters[videoId] || []);
          const visible = allChapters.filter(c => c.start < facadeEnd && c.end > facadeStart);
          if (visible.length === 0) return;
          const ul = document.createElement('ul');
          ul.className = 'video-chapters';
          ul.setAttribute('aria-label', 'Chapters in this segment');
          visible.forEach(c => {
            const seek = Math.max(c.start, facadeStart);
            const mm = String(Math.floor(seek / 60)).padStart(1, '0');
            const ss = String(seek % 60).padStart(2, '0');
            const li = document.createElement('li');
            const btn = document.createElement('button');
            btn.type = 'button';
            btn.className = 'video-chapter-btn';
            btn.dataset.seek = seek;
            btn.textContent = `${mm}:${ss} — ${c.title}`;
            li.appendChild(btn);
            ul.appendChild(li);
          });
          facade.insertAdjacentElement('afterend', ul);
        });
      }

## Known risks + mitigations

1. **CLS min-height over-reservation** — if a facade has only 1 chapter, the `min-height: 80px` reserves space for 3. Slightly more whitespace than ideal. Tradeoff accepted for CLS-zero.
2. **DOM insertion before iframe activation** — `insertAdjacentElement('afterend', ul)` places the UL immediately after the facade `<button>`. When facade → iframe swap happens via `replaceWith`, the iframe takes the facade's place but the UL remains as a sibling (still after the iframe). Design intent.
3. **Chapter title overflow on mobile** — long titles wrap. No truncation applied; CSS `text-align: left` + natural wrapping handles this. If ugly, operator flags during C5.5 verify.
4. **Chapter button focus order** — Tab order is facade → chapter1 → chapter2 → ... → next facade. Readers can Tab through to skip the chapters. Acceptable.
5. **Empty chapter list** — the code's `if (visible.length === 0) return;` prevents empty UL elements. Given the measured counts in C5.1, this path shouldn't fire, but is defensive.
